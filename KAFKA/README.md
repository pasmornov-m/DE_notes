# APACHE KAFKA

---

## Введение

Kafka - система, которая даёт возможность публиковать потоки данных и подписываться на них, а также хранить данные. Данные в Kafka рассматриваются как непрерывно развивающийся и постоянно растуший поток. Это распределённая кластерная система, способная масштабироваться горизонтально по мере роста потребностей. Kafka способна хранить данные столько времени, сколько потребуется, читать данные можно в любое время, с любого момента в прошлом. Также, присутствует репликация данных, гарантия доставки сообщений, целостность. Представляет собой JAVA приложение, что обеспечивает работу на различных ОС. 

---

## Основные концепции

### Сообщение

Это минимальная единица данных, по аналогии со строкой записи в БД. Сообщение с точки зрения Kafka сообщение - массив байтов. В сообщении может содержаться необязательный элемент метаданных - ключ (тоже массив байтов), для Kafka он не несёт никакого смысла, но от него зависит распределение по партициям при записи. Данные записываются и читаются пакетами (batch) для эффективности, чтобы не передавать отдельные сообщения по сети.

### Топики и партиции (topics & partitions)
Темы и разделы

Сообщения распределяются по топикам (topic) - по аналогии с таблицами в БД. Топики, в свою очередь, разделяются на партиции (partition). Сообщения добавляются в конец партиции, а читаются от начала к концу. Kafka гарантирует упорядоченность сообщений только в пределах одной партиции. Благодаря партициям обеспечивается избыточность (репликация) и масштабируемость - разделы можно разместить на разных брокерах. Offset - целочисленный номер сообщения в партиции, который непрерывно увеличивается, а Kafka записывает его в каждое сообщение. 

### Producer

Producer записывает сообщение в топик. По умолчанию он записывает соббщения равномерно по партициям. Сообщения с одним и тем ключом попадают в одну партицию. Продюсер может ипользовать свой объект Partitioner, чтобы распределять сообщения по темам в любом нужном порядке. 

### Consumer

Консьюмеры подписываются на топик и читают сообщения в порядке их создания. Благодаря тому, что offset последнего прочитанного консьюмером сообщения записывается в zookeeper или в kafka, консьюмер может останавливаться и возобновлять работу. Консьюмеры работают в группе (consumer group) - один или несколько консьюмеров, читающих одну патрицию совместно. Организация в группу гарантирует чтение каждого раздела только одним членом группы. При сбое одного члена группы его раздел будет читать другой с первого непрочитанного offset (смещения). 

### Consumer group

Групп может быть произвольное количество. Каждая группа потребителей читает в своём темпе и порядке, как будто других групп потребителей не существует. Для каждой группы сохраняется последний прочитанный offset топика при помощи zookeeper или самой kafka. Увеличение членов группы позволет ускорять потребление и горизонтально масштабироваться. Но не более чем партиций в топике. Для топика можно увеличить число партиций, но не уменьшать их. Консьюмеры, которым не хватило партиции будут простаивать, но могут подключиться при сбое работающего консьюмера из группы. 

### Основные параметры конфигурации брокера Kafka

1) broker id - целочисленный идентификатор уникальный в пределах кластера
2) port - по умолчанию 9092, можно изменить
3) log dirs - путь, по которому kafka сохраняет сообщения
4) num partitions - определяет, с каким числом партиций создаётся топик
5) log retention ms - продолжительность хранения сообщения в Kafka
6) log retention bytes - размер сохраняемых сообщений в байтах
7) message max bytes - максимальный размер генерируемых сообщений

### Семантики доставки сообщений и подтверждения об успешной записи

За сохранность сообщений в Kafka отвечают семантики доставки сообщений.

1) хотя бы один раз (at least once) - отправитель сообщения получает подтверждение от брокера Kafka, `acks = all`, что гарантирует однократную запись сообщения в топик с повторением отправки при отсутствии подтверждения, при этом сообщение может быть дублировано.
2) не более одного раза (at most once) - отправитель не повторяет отправку при отсутствии подтверждения, при этом возможно, что сообщение не записано в топик.
3) строго однократно (exactly once) - при повторной попытке отправителя отправить сообщение оно доставляется строго один раз, дублирование отсутствует. Для этого следует задать свойство идемпотентности в настройках отправителя. 

Параметр acks влияет на надёжность и может принимать следующие значения:

- **0**, когда продюсер вообще не ждёт никакого подтверждения. Невозможно гарантировать, что сервер получил запись. 

- **1**, не ждёт полного подтверждения от всех сервров (подписчиков, followers). Сообщение может быть потеряно.

- **-1** или **all**, когда продюсер ждёт полной репликации сообщения по всем серверам кластера, что обеспечивает надёжную защиту от потери данных, но увеличивает задержку (latency) и снижает пропускную способность.

## Архитектура

### Брокеры и кластер

Отдельный сервер Kafka называется брокер. Данные храняться на локальном диске брокера. Один из работающих брокеров автоматически выбирается в качестве контроллера, который отвечает за административные функции по всему кластеру: распределение партиций по брокерам, мониторинг отказов. Каждая партиция принадлежит одному брокеру, который называет ведущим (leader). Можно установить фактор репликации для топика так, что разделы будут реплицироваться на другие брокеры. При сбое лидера его место занимает другой брокер, содержащий его реплику. Все продюсеры и консьюмеры, работающие с разделом, работают только с лидером. 

### Хранение сообщений

Основная особенность - хранение информации в течение длительного времени. Для каждого топика можно установить хранение информации в течение промежутка времени (например 7 дней) или по достижении топиком определённого размера в байтах (например 1 ГБ). После превышения ограничения более старые данные удаляются. Для топика можно настроить вариант хранения - сжатый журнал (log compacted). При этом будет сохраняться лишь последнее сообщение для данного ключа, пример - журнал изменений, в случае, если нас интересует только последнее состояние без исторических данных.

### Kafka и Zookeeper

Zookeeper используется для хранения метаданных кластера, также используется для хранения текущих offset для потребителей. Zookeeper обновляется редко, поэтому для кластера можно использовать уже установленный общий ZK. Т.к. читатели записывают в ZK информацию о смещении с определённым интервалом времени, это может стать слабым местом в больших кластерах, поэтому в последних версиях Kafka информацию о смещении клиент может сохранять непосредственно в самой Kafka, снижая нагрузку на ZK.

### Схемы сообщений

Для Kafka - это просто набор байтов. Для того, чтобы согласовать схему данных для продюсера и консьюмера есть несколько способов. Самый простой вариант - использование JSON или XML, т.к. это просто текст вместе с метаданными, но такой подход имеет низкую устойчивость к ошибкам по типам данных и не имеет совместимости различных весрий схем. Другой вариант - Apache Avro, который является фреймворком сериализации, обеспечивающий компактный формат хранения и типизацию данных, имеющий совместимость схем разных версий. В файлах Avro схема храниться в самиом файле. Но не эффективно передавать схему (JSON) вместе с каждым сообщением, т.к. это увеличивает объём сообщения. Версиий схем хранятся в общем реестре схем (Schema Registry), куда могут записывать продюсеры и читать консьюмеры. В сообщении Kafka храниться только id схемы и консьюмер может её считать из общего реестра.
