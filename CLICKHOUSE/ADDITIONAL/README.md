# Полезные функции ClickHouse

# 1. sequenceMatch и sequenceCount — анализ последовательностей событий

## Что делают

Позволяют находить **последовательности событий пользователя** без сложных JOIN и подзапросов.

## Основная идея

Вместо соединения таблицы самой с собой задаётся **шаблон событий**, который должен выполниться в заданном порядке.

Пример логики:

```
view_product → click_buy → purchase
```

Можно задать ограничение по времени между шагами.

### Преимущества

* не требуется self-join
* данные читаются один раз
* код становится компактнее
* высокая производительность на больших объёмах

### sequenceCount

Похож на `sequenceMatch`, но возвращает:

```
количество найденных последовательностей
```

Полезно для анализа повторных конверсий.

---

# 2. windowFunnel — построение воронок

## Что делает

Определяет:

* до какого шага воронки дошёл пользователь
* сколько пользователей остановилось на каждом шаге

## Как работает

Задаётся:

```
windowFunnel(window)(timestamp, step1, step2, ...)
```

Где `window` — общее время на прохождение всей воронки.

### Возвращает

```
номер последнего выполненного шага
```

Пример:

```
0 — только просмотр
1 — начал покупку
3 — завершил покупку
```

### Когда использовать

* анализ продуктовых воронок
* расчёт конверсий
* анализ поведения пользователей

# 3. retention() — когортный retention одной функцией

## Что делает

Позволяет считать **retention по периодам** (например D1, D7, D30).

Функция принимает набор условий:

```
day_0
day_1
day_7
day_30
```

И возвращает массив:

```
[1,1,0,0]
```

### Логика

Если:

```
cond1 выполнено → r[1] = 1  
cond1 и cond2 выполнены → r[2] = 1  
```

### Преимущества

* уменьшает риск логических ошибок
* упрощает код
* удобно работать с когортами
* сразу возвращает массив результатов

### Когда использовать

* retention-анализ
* когортные исследования
* анализ возвращаемости пользователей

# 4. Комбинаторы агрегатных функций

ClickHouse позволяет добавлять к агрегатным функциям специальные суффиксы:

```
-If
-Distinct
-State
-Merge
-ForEach
```

Это значительно упрощает SQL.

---

## 4.1 -If — условные агрегаты

Позволяет писать условные агрегаты без `CASE`.

Пример логики:

```
countIf(os = 'iOS')
countIf(os = 'Android')
```

### Преимущества

* меньше подзапросов
* чище код
* быстрее выполнение

Работает с:

```
countIf
sumIf
avgIf
uniqIf
anyIf
```

---

## 4.2 -Distinct — агрегаты по уникальным значениям

Позволяет убрать `SELECT DISTINCT` из подзапросов.

Примеры:

```
avgDistinct(amount)
sumDistinct(amount)
avgDistinctIf(amount, cond)
```

### Преимущества

* упрощает код
* уменьшает вложенность запросов
* снижает вероятность ошибок

---

## 4.3 -ForEach — агрегаты по массивам

Позволяет обрабатывать массивы как отдельные колонки.

Полезно при работе с:

* массивами метрик
* массивами категорий
* фиксированными структурами данных

### Важно

Работает по **позициям массива**, а не по значениям.

Подходит для:

```
[views, clicks, cart, purchases]
```

Но не подходит для:

```
списка произвольных категорий
```

(там нужен `arrayJoin`).

---

# 5. cityHash64() — мощный инструмент оптимизации

## Что делает

Преобразует данные:

```
строки / UUID / массивы → 64-битное число
```

Основная идея:

```
меньше байт → быстрее обработка → меньше памяти
```

---

# Когда использовать cityHash64()

Подходит для:

* длинных строк
* URL
* cookies
* UUID
* массивов
* больших ключей JOIN

---

# Основные применения cityHash64()

---

## 5.1 Оптимизация uniq()

Вместо:

```
uniqExact(s)
```

Можно использовать:

```
uniqExact(cityHash64(s))
```

### Эффект

* уменьшение памяти
* ускорение выполнения
* примерно ×2 по памяти

---

## 5.2 Оптимизация IN

Вместо сравнения строк:

```
s IN (...)
```

Используется:

```
cityHash64(s) IN (...)
```

### Эффект

* до ×4 ускорения
* уменьшение памяти

---

## 5.3 Оптимизация JOIN

Можно джойнить:

```
по hash вместо строки
```

Если исходное поле тяжёлое.

### Эффект

* ускорение
* уменьшение RAM

---

## 5.4 Разбиение данных на батчи

Используется:

```
cityHash64(s) % N
```

Позволяет:

* равномерно разбить данные
* выполнять расчёты частями
* избежать Memory Limit

Полезно для:

* больших аналитических расчётов
* тяжёлых агрегаций

---

## 5.5 Хранение hash-колонок

Можно заранее сохранить:

```
s_hash UInt64 DEFAULT cityHash64(s)
```

Преимущества:

* меньше чтения с диска
* быстрее GROUP BY
* быстрее uniq()

Иногда ускорение достигает:

```
×100 и более
```

---

## 5.6 Оптимизация GROUP BY

Вместо:

```
GROUP BY s
```

Используется:

```
GROUP BY cityHash64(s)
```

Или:

```
GROUP BY s_hash
```

### Эффект

* сильное уменьшение памяти
* значительное ускорение

---

## 5.7 Работа с несколькими колонками

Используется:

```
cityHash64((c1, c2))
```

Важно:

Использовать **tuple**, иначе `NULL` ломает результат.

---

## 5.8 Валидация данных

Можно использовать:

```
groupBitXor(cityHash64(tuple(*)))
```

Это работает как:

```
checksum таблицы
```

Полезно при:

* миграциях
* сравнении данных
* проверке корректности

---

## 5.9 Воспроизводимая случайная выборка

Вместо:

```
ORDER BY rand()
```

Используется:

```
ORDER BY cityHash64(...)
```

Результат:

* случайный
* но воспроизводимый

Очень полезно для:

* тестирования
* отладки
* сравнения результатов
